# CGS Iterative Refinement — Investigation Notes

## Context

The single-pass eager CGS adaptation (`ChoiceGradientSampling.swift`) successfully biases pick weights toward predicate satisfaction, but collapses diversity on recursive generators like BSTs. This document records findings and outlines avenues for iterative refinement.

## Benchmark: Valid BSTs (height >= 1, `isValidBST()`)

Generator: `BinarySearchTree<UInt>.arbitrary` — recursive pick (leaf vs node) with max depth 5, values in 0...9.

### Results (1-minute runs, M1 MacBook Pro)

| Method | Total valid | Unique valid | Height distribution |
|--------|------------|-------------|---------------------|
| Paper — Rejection | 7,354 | 109 | Not reported |
| Paper — CGS | 22,107 | 338 | Not reported |
| Exhaust — Rejection | 110,729 | 923 | h1: 92,977 h2: 15,988 h3: 1,586 h4: 92 h5: 86 |
| Exhaust — CGS (single pass) | 12,353,098 | **10** | h1: 12,353,098 |

### Observations

1. **Throughput**: CGS is ~100x faster than rejection (12M vs 110K values/min) because every generated value is valid. But all 10 unique values are `node(leaf, v, leaf)` for v in 0...9.

2. **Diversity collapse**: At each recursive pick (leaf vs node), the adaptation correctly measures that the leaf branch produces valid subtrees 100% of the time while the node branch has a ~4% validity rate at the top level, dropping further at depth. The single pass crushes deeper node branches to near-zero weight.

3. **Composed predicate signal**: The composed predicate at depth D must sample through D layers of continuation to evaluate. At depth 2+, random sampling almost never produces a valid BST by chance, so all deeper node branches get weight 0 and fall back to equal weights — but the pick at depth 1 already chose leaf.

4. **Why the paper does better**: The paper likely uses iterative refinement — multiple adaptation passes where each iteration uses the *previously adapted* generator for sampling. This bootstraps signal deeper:
   - Pass 1: Learns that height-1 trees are mostly valid. Biases toward them.
   - Pass 2: Since the adapted generator produces valid subtrees more often, sampling through the continuation at depth 2 now has higher success rates. Deeper node branches get nonzero weights.
   - Pass N: Signal propagates one level deeper per pass.

## Proposed approach: multi-pass adaptation

```
func adaptIteratively(generator, predicate, passes: Int) {
    var current = generator
    for _ in 0..<passes {
        current = adapt(current, predicate: predicate)
    }
    return current
}
```

### Key questions

1. **Does re-adapting an already-adapted generator work mechanically?** The adapted generator has synthesised picks (from chooseBits subdivision) with specific weights. Re-adaptation would re-sample and re-weight those picks. The inner structure should be compatible — it's still a `ReflectiveGenerator` with the same operation types.

2. **Does signal actually propagate deeper?** After pass 1, the adapted generator produces height-1 trees much more often. When pass 2 samples through the continuation at depth 2, the subtrees generated by the continuation are now biased toward validity. This should give the node branch at depth 2 a nonzero success rate, pulling signal one level deeper.

3. **How many passes are needed?** For a BST with max depth 5, we'd expect ~5 passes to propagate signal to the deepest level. The paper uses 15 iterations, which suggests either: (a) more passes are needed for convergence, or (b) they use a different strategy (e.g. gradient-like weight updates rather than full replacement).

4. **Convergence detection**: The paper reports "convergence iterations". We could detect convergence by comparing the weight vectors between passes — if they stabilise (within some epsilon), stop early.

5. **Sample count per pass**: The paper uses N=50 samples. With iterative refinement, each pass can use fewer samples since it's building on the previous pass's structure. The total oracle budget is `passes * samples_per_pass`.

6. **Weight accumulation vs replacement**: Single-pass CGS replaces weights entirely. An alternative is to *blend* old and new weights: `new_weight = alpha * sampled_weight + (1 - alpha) * old_weight`. This would be more conservative and might avoid over-fitting to easy branches.

## Alternative approaches

### Bottom-up adaptation
Instead of top-down with composed predicates, adapt from the leaves up:
1. At the deepest picks, adapt using a local predicate (e.g. "this value fits within what range?").
2. Move up one level, using the already-adapted subtree generators for sampling.
3. Repeat to the root.

This naturally propagates signal from leaves to root and avoids the composed predicate's sampling problem. However, it requires knowing the predicate decomposition at each level, which contradicts the paper's black-box approach.

### Continuation-aware sampling
When sampling through a composed predicate, instead of using `ValueInterpreter.generate` (which generates randomly), use the *adapted* version of the continuation's generators. This requires making the adaptation context aware of previously adapted sub-generators — essentially interleaving adaptation and evaluation.

### Depth-proportional sample budgets
The current exponential decay (`baseSamples / 2^depth`) reduces samples at depth. For iterative refinement, the opposite might help: deeper picks need *more* samples in early passes because the signal is weaker. Budget could be `baseSamples * 2^(pass - depth)` — more samples at depth in early passes, fewer in later passes when the signal is stronger.
